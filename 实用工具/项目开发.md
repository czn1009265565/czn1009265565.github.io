# 项目开发
## 一、需求分解方法
### 分层拆解
按「业务目标→功能模块→子功能→具体任务」层级拆分

- 顶层先对齐核心业务目标，避免偏离需求本质
- 中层拆分为独立功能模块，比如电商系统可拆为用户模块、商品模块、订单模块等
- 底层拆解为可单人执行的具体任务，每个任务颗粒度控制在1~3人日可完成

### 明确边界
每个拆解后的任务标注「输入、输出、验收标准、依赖项」，避免模糊需求导致的后期返工
### 分类筛选
将需求划分为「核心必需、优化增强、可选扩展」三类，优先保障核心需求不被非必要功能挤占资源

## 二、开发顺序制定原则
1. 依赖优先先开发基础支撑能力（比如用户鉴权、数据库表设计），再开发上层业务功能，避免阻塞后续任务。
2. 核心先行第一优先级排期核心必需类需求，先保障主流程可用，再迭代优化细节。
3. 风险前置把技术难度高、不确定性强的任务放在前期开发，提前暴露问题、预留调整时间。
4. 可验证性尽量按独立业务闭环排期，每完成一个模块即可单独测试验证，避免所有功能耦合到最后才集中联调。如果是敏捷开发模式，可按2~4周为一个迭代周期，每个周期交付一个可运行的功能版本，持续对齐需求。
